Часть 1

В качестве экспериментов использовал простое консольное приложение на go

Bad practices в CICD файле cicd_configs/bad_cicd.yaml:
    
1. Запуск пайплайна на pull request на каждую ветку - лишние запуски пайплаинов

2. Несколько разных джоб скомбинированы в одну сложную - сложно дебажить и легко допустить ошибки

3. Игнорирование зафейлиных тестов - а вот и пример ошибок, возможно была временная заглушка, которую забыли убрать

4. Отключение кэша - пайплайн будет работать дольше.

5. Отсутствие Environment - секреты берутся из общих настроек репозитория

6. Вывод секретов в логи (печатается ***, так как github маскирует) - нарушение безопасности

7. Использование старых версий actions или latest - что-то может некорректно работать

8. Отсутствие старых версий uses или вообще без указания версии - что-то может некорректно работать

9. Создание бинарного файла go без флагов - бинарник весит значительно больше и содержит ненужную инфу

10. Deploy на всех ветках - вряд ли нужно деплоить, например, дев

<br>

Исправленный CICD файл cicd_configs/good_cicd.yaml (в .github/workflows/hw4_cicd.yaml - копия):

1. Запуск пайплайна на pull request только на ветку main - нет лишних запусков

2. Джобы разделены по назначению - проще дебажить, а так же установлены зависмости (needs), чтобы не запускать все джобы, если на предыдущих что-то сломалось

3. Игнорирование зафейлиных тестов - тесты проверяются, все ок

4. Кэши включены - пайплайны будут работать быстрее, так как не нужно будет весзде скачивать зависимости. (go.mod меняется не часто)

5. Добавлен Environment - секреты берутся из заданной среды

6. Для проверки секрета выводится его длина - безопасно

7. Использование актульных stable версий actions - минимизированы возможные ошибки

8. Отсутствие старых версий uses или вообще без указания версии - минимизированы возможные ошибки

9. Создание бинарного файла go с флагами - бинарник весит меньше, не содержит ненужную инфу

10. Deploy только на ветке main - ожидаемый результат, просто добавлен кондишн

Часть 2 - как я страдал с vault

1. Нужно поднять у себя сервис с vault.

Это в целом несложно, простой docker compose файл легко найти. Но есть нюанс - в примерах для начала рекомендуется запуск в режиме разработчика (-dev).
В dev-режиме Vault хранит всё в оперативной памяти, в целом, ничего настраивать особо не требуется.
Для учебной задачи это ок, но не для меня, я не ищу легких путей.

2. Нужно поднять у себя сервис с vault с сохранением данных на диск

И это в целом тоже несложно, если конечно не в первый раз делаешь.

- Немного изменить docker compose файл, в частности: command: server -config=/vault/config/vault.hcl

4. А как тепрь синтегрировать с пайплайном?

Интеграция с Vault сделана, секреты на этапе деплоя подтягиваются из поднятого авторизованного хранилища, а не хранятся в CI/CD переменных репозитория.

<br>

Почему хранение секретов в переменных CI/CD плохо?

1. Нужно менять секреты вручную, что может быть очень накладно, особенно если секретов много.

2. Доступны кому надо, и кому не надо.

3. Секреты в GitHub лежат вечно. В Vault можно настроить ttl для секреты.

<br>

Почему мой способ «красивый»?

1. Секрет не хранится в переменных CI/CD, а подтягивается из хранилища, секрет зашифрованы и хранится изолированно.

2. Безопасная проверка - печатается длина, а не секрет.

3. Использование vault-action - секрет автоматически маскируется (***) и вычищается после завершения джобы

4. Самое главное - я все же поднял у себя полноценный vault c сохранение секретов на диске.
